# Contributing to Librelancer

We're happy to see you're interested in contributing to Librelancer, whether it be through issues or pull requests. Please keep in mind a few key guidelines when preparing to contribute:

1. **Submissions should be your own work**

Librelancer does not accept issues or pull requests where it is evident that the main author of the description, comments, or code is an LLM or AI-assisted tool. **Issues and PRs where LLM-isms are evident will be closed automatically.**

Code must also be able to be licensed under the MIT license, and cannot come from other projects with more restricitive license terms.

2. **Check against vanilla Freelancer, and don't assume it is logical**

Freelancer has a particularly quirky game engine, and all of us have fallen into the trap of thinking certain strings or functions behave in a way that would be logically consistent, particularly in a modern context where a lot of these features are common across games.

Freelancer's approach to several things can be unorthodox, and it is best to check behaviour with the original game before committing.

3. **Be concise**

Where possible, try to keep PRs touching one thing at a time. Sometimes certain bugfixes can be merged immediately while others need extra work. Keeping these separate means more work can be merged faster.

4. **Keep in touch**

When it comes to larger changesets, it is best to have a chat before opening a PR. This makes reviews easier, and helps make sure a change is something that can be upstreamable and in the right direction before a big set of changes is to be reviewed.
